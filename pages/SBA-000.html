Introduction

First of all, it is my intent to have a static website in the permaweb. Therefore, it will function as my permanent archive that will never go down (supposedly). Because of my lack in technical expertises, I am going to use Pandoc as my html parser.

However, to do so, we have to have some interactive elements on it. Namely, a navbar atop of the document or blog articles. To do so, I would also need to learn a bit of css to make unordered list item listed inline. Because we need to understand a bit of CSS, we have to also know about CSS selector.

Then it comes with another feature (and challenge) of the permaweb: it is immutable. Therefore, updating would be a hassle, and we need to get creative. Hence the need to learn about how to include html snippets in html.

We should also consider to make the web responsive. Apparently we can do it by having a responsive navigation bar.

Primary Sources

Pandoc

Pandoc is a very useful parser, especially for a plain html file. However, it will output only a normal standalone page with very plain appearance. To make some elements work, here are some important parts of the documentation that may be of interest.

Include files

Source: Pandoc, General writer options

-H FILE, –include-in-header=FILE|URL

Include contents of FILE, verbatim, at the end of the header. This can be used, for example, to include special CSS or JavaScript in HTML documents. This option can be used repeatedly to include multiple files in the header. They will be included in the order specified. Implies --standalone.

-B FILE, –include-before-body=FILE|URL

Include contents of FILE, verbatim, at the beginning of the document body (e.g. after the <body> tag in HTML, or the \begin{document} command in LaTeX). This can be used to include navigation bars or banners in HTML documents. This option can be used repeatedly to include multiple files. They will be included in the order specified. Implies --standalone.

-A FILE, –include-after-body=FILE|URL

Include contents of FILE, verbatim, at the end of the document body (before the </body> tag in HTML, or the \end{document} command in LaTeX). This option can be used repeatedly to include multiple files. They will be included in the order specified. Implies --standalone.

Mathjax

Source: Pandoc, Math rendering in HTML.

The default is to render TeX math as far as possible using Unicode characters. Formulas are put inside a span with class="math", so that they may be styled differently from the surrounding text if needed. However, this gives acceptable results only for basic math, usually you will want to use --mathjax or another of the following options.

--mathjax[=URL]

Use MathJax to display embedded TeX math in HTML output. TeX math will be put between \(...\) (for inline math) or \[...\] (for display math) and wrapped in <span> tags with class math. Then the MathJax JavaScript will render it. The URL should point to the MathJax.js load script. If a URL is not provided, a link to the Cloudflare CDN will be inserted.

Divs and Spans

It is apparently possible to also include <div> and <span> elements in pandoc. Might be helpful so we don’t have to generate the navbar from scratch with pure html. Perhaps then we can type in the navbar in pandoc before parsing it to a html snippet.

Apparently it uses fenced_divs extension of pandoc, and attributes can be included in a pair of curly braces, or a single unbraced word that would act as a class name. The div element terminates with a string of at least three consecutive colons.

Examples:

Example:

::::: {#special .sidebar}
Here is a paragraph.

And another.
:::::

Fenced divs can be nested. Opening fences are distinguished because they must have attributes:

::: Warning ::::::
This is a warning.

::: Danger
This is a warning within a warning.
:::
::::::::::::::::::

Fences without attributes are always closing fences, though they don’t have to be of the same length.

fenced_code_attributes (Edit 20220905)

This section is added far after the page is created. The original version uses only fenced_code_blocks with backtick_code_blocks that results only in plain monospace codeblocks. Just recently I learned that we can make it prettier (showing properly highlighted code blocks) with the use of fenced_code_attributes. The difference will be demonstrated below:

<video width="320" height="240" controls>
  <source src="../videos/20200723_172326.mp4" type="video/mp4">
Your browser does not support the video tag.
</video>

The block above uses ordinary fenced_code_blocks, and will result in a plain monoscape code block. Basically we use backticks (`) to wrap our code block, and it appears like the following in our source file:

```
<... a html code block ...>
```

However, we can instead do the following:

<video width="320" height="240" controls>
  <source src="../videos/20200723_172326.mp4" type="video/mp4">
Your browser does not support the video tag.
</video>

That uses the following syntax to wrap our code block:

```html
<... a html code block ...>
```

If you are a first time visitor, then congratulations! You are seeing beautifully highlighted code blocks. If you’ve seen the previous version of this page, then you’re welcome! Now you can easily follow through the codes without having an eyesore.

A Decent Pandoc Template With Sidebar and Nav

I just discovered this one template recently. Probably I wouldn’t use it for the first blog, but I will just put it here for later reference.

Navbar

I begin my search here, that apparently comes from an era very long time ago. It was from 2007, and apparently today it is supposedly much more easier.

Update April 2013: This article is pretty old. This isn’t very hard. Just give the list centered text (e.g. ul.nav { text-align: center; }) and the list items inline-block (e.g. ul.nav li { display: inline-block; }). If you want to do it with margin for whatever reason, look into width: fit-content;.

Then, I found a working example from w3schools.com. The basic premise is that navigation bar is just a list of links. It can be listed vertically or horizontally, and needs a html base. Therefore the article uses <ul> and <li> elements.

For example:

<ul>
  <li><a href="default.asp">Home</a></li>
  <li><a href="news.asp">News</a></li>
  <li><a href="contact.asp">Contact</a></li>
  <li><a href="about.asp">About</a></li>
</ul>

Then add some css to remove the bullets and the margins and padding from the list:

ul {
  list-style-type: none;
  margin: 0;
  padding: 0;
}

The list-style-type: none; part removes list markers (or bullets) from an unordered list. The later parts: margin: 0; and padding: 0; are to remove browser default settings.

Vertical Navbar

After we built our basic navbar elements, we can style the <a> elements inside the list, that is what happens when we hover our cursor above a list element:

li a {
  display: block;
  width: 60px;
}

The display:block; part is used to make the entire link area clickable instead of just the text. It also makes specifying the width, padding, margin, height, etc, possible. It is demonstrated in the width: 60px; section, where we make our list item 60 pixels wide, instead of the default 100%.

Alternately, instead of specifying the width on <a> elements, we specify it on <ul> element. It will produce the same results as above, and coded something like this:

ul {
  list-style-type: none;
  margin: 0;
  padding: 0;
  width: 60px;
}

li a {
  display: block;
}

Then we can construct a basic vertical navigation bar with a gray background color. On hovering, we want the background color of the links to change.

The styling is as follows:

ul {
  list-style-type: none;
  margin: 0;
  padding: 0;
  width: 200px;
  background-color: #f1f1f1;
}

li a {
  display: block;
  color: #000;
  padding: 8px 16px;
  text-decoration: none;
}

/* Change the link color on hover */
li a:hover {
  background-color: #555;
  color: white;
}

Apparently, it is possible to have an “active” class to the current link, so the user know which page they are on:

.active {
  background-color: #04AA6D;
  color: white;
}

However I am not sure how to implement that. I suspect that it is done on the html part, like the following:

<ul>
  <li><a href="default.asp">Home</a></li>
  <li><a href="news.asp" class="active">News</a></li>
  <li><a href="contact.asp">Contact</a></li>
  <li><a href="about.asp">About</a></li>
</ul>

Note that the “News” list item contains a class=“active” attribute.

We can also center links and add borders to the navbar. We can center the links by having text-align:center to <li> or <a>. To add borders to your navbar, you can use the border property on <ul>. To add borders inside the navbar, add a border-bottom to all <li> elements, except for the last one.

Here is an example styling definitions:

ul {
  border: 1px solid #555;
}

li {
  text-align: center;
  border-bottom: 1px solid #555;
}

li:last-child {
  border-bottom: none;
}

We can also have a full-height fixed vertical bar, or a “sticky” side navigation. Note that it might not work properly on mobile devices. To do so, we use the following styling definitions:

ul {
  list-style-type: none;
  margin: 0;
  padding: 0;
  width: 25%;
  background-color: #f1f1f1;
  height: 100%; /* Full height */
  position: fixed; /* Make it stick, even on scroll */
  overflow: auto; /* Enable scrolling if the sidenav has too much content */
}

Horizontal Navbar

This is the part we are interested at: a horizontal navbar! Apparently, there are two ways to create a horizontal navbar. The first method is through the use of inline list items, while the second method uses the floating list items.

Inline list items for a horizontal navbar can be done by first specify the <li> elements as inline, in addition to the “standard” code from the previous section:

li {
  display: inline;
}

By default, <li> elements are block elements. The display: inline; removes line breaks before and after each list item, hence they are all displayed as a single line.

Floating list items for a horizontal navbar can be done by floating the <li> elements, and specify a layout for the navigation links:

li {
  float: left;
}

a {
  display: block;
  padding: 8px;
  background-color: #dddddd;
}

The float: left; directive uses float to get block elements to float next to each other. Then with display: block;, it allows us to specify padding, height, width, margin, etc. to the block. In the above code snippet, we use padding: 8px directive to specify a 8 pixel padding between each <a> element. The background-color: #dddddd; adds a gray background color to each <a> element.

Alternatively, you can add the background-color to <ul> instead of each <a> element if you want a full-width background color:

ul {
  background-color: #dddddd;
}

Let us have some examples to showcase possibilities of displaying navbars.

The first example uses a basic horizontal navbar with a dark background color, and change the background color of the links when the user moves the mouse over them:

ul {
  list-style-type: none;
  margin: 0;
  padding: 0;
  overflow: hidden;
  background-color: #333;
}

li {
  float: left;
}

li a {
  display: block;
  color: white;
  text-align: center;
  padding: 14px 16px;
  text-decoration: none;
}

/* Change the link color to #111 (black) on hover */
li a:hover {
  background-color: #111;
}

We can also add an “active” class to the current link to let the user know which page they are on:

.active {
  background-color: #04AA6D;
}

We can also right-align links by floating the list items to the right ( float:right; ):

<ul>
  <li><a href="#home">Home</a></li>
  <li><a href="#news">News</a></li>
  <li><a href="#contact">Contact</a></li>
  <li style="float:right"><a class="active" href="#about">About</a></li>
</ul>

We can add the border-right property to <li> to create link dividers:

/* Add a gray right border to all list items, except the last item (last-child) */
li {
  border-right: 1px solid #bbb;
}

li:last-child {
  border-right: none;
}

To make the navigation bar to stick at the top or bottom of the page, we can specify:

Fixed top:

ul {
  position: fixed;
  top: 0;
  width: 100%;
}

Fixed bottom:

ul {
  position: fixed;
  bottom: 0;
  width: 100%;
}

Note that fixed position might not work properly on mobile devices.

To have a gray horizontal bar with a thin gray border, with a thin gray border, we specify:

ul {
  border: 1px solid #e7e7e7;
  background-color: #f3f3f3;
}

li a {
  color: #666;
}

We can also add position: sticky; to <ul> to create a sticky navbar. A sticky element toggles between relative and fixed. The toggle depends on the scroll position. It is positioned relative until a given offset position is met in the viewport. After the condition is met, the element sticks in place like position:fixed.

ul {
  position: -webkit-sticky; /* Safari */
  position: sticky;
  top: 0;
}

Update 20220921: Our navbar can sometimes get overwritten with styling rules of highlighted code blocks, or even video elements. Therefore a solution is needed: z-index values. Basically I modified the navbar styling to the following:

ul.nav {
  list-style-type: none;
  margin: 0;
  padding: 0;
  overflow: hidden;
  background-color: rgb(24, 25, 26);
  position: fixed;
  top: 0;
  left: 0;
  width: 100%;
  z-index: 1000; /* This is the fix */
}

Also, our navbar sometimes hide headings when they’re selected from our table of contents. To fix that, we followed a short trick from css-tricks.com, and it works like magic! Basically we have to specify the scroll-margin-top value. In our case we use 2.5rem. Our heading styling rules are now as follows:

h1, h2, h3, h4, h5, h6 {
  margin-top: 1.4em;
  /* below is the fix */
  scroll-margin-top: 2.5rem;
}

CSS Selector

From w3schools.com, we learn about CSS selectors. Apparently, here’s a quick cheatsheet for the most basic CSS selectors:

Selector Example Example description
#id #firstname Selects the element with id=“firstname”
.class .intro Selects all elements with class=“intro”
element.class p.intro Selects only <p> elements with class=“intro”
* * Selects all elements
element p Selects all <p> elements
element,element,.. div, p Selects all <div> elements and all <p> elements

The following is a basic explanation:

The CSS element selector selects HTML element of the same name. For example, to select <p> element, we use p CSS selector:

p {
  text-align: center;
  color: red;
}

The CSS id selector selects the id attribute of a HTML element. As the id of an element is unique within a page, the selector is used to select one unique element. In CSS, we use a hash (#) character, followed by the id of the element. For example, to select a HTML element with id=“para1”,1 we use:

#para1 {
  text-align: center;
  color: red;
}

The CSS class selector selects HTML elements with a specific class attribute, with the use of a period (.) character, followed by the class name.

For example, if we want to find all HTML elements with class=“center”, we type:

.center {
  text-align: center;
  color: red;
}

It is also possible to select a specific HTML elements with a certain class. For example, if we want a style to affect only <p> elements with class=“center”, we type:

p.center {
  text-align: center;
  color: red;
}

Do note that HTML elements can also refer to more than one class:

<p class="center large">This paragraph refers to two classes.</p>

Note that a class name cannot start with a number.

We can also universally select all HTML elements in a page with an asterisk (*) character:

* {
  text-align: center;
  color: blue;
}

Alternately, we can group select specified elements that share the same style definitions. So instead of typing:

h1 {
  text-align: center;
  color: red;
}

h2 {
  text-align: center;
  color: red;
}

p {
  text-align: center;
  color: red;
}

We can type all of the affected elements, with each selectors separated by a comma:

h1, h2, p {
  text-align: center;
  color: red;
}

Style HR

HR (horizontal line) is a html element that is usually associated with a thematic change of contents. As an element in html, it can also be styled with CSS. To style a hr element, we use its border property:

/* Red border */
hr.new1 {
  border-top: 1px solid red;
}

/* Dashed red border */
hr.new2 {
  border-top: 1px dashed red;
}

/* Dotted red border */
hr.new3 {
  border-top: 1px dotted red;
}

/* Thick red border */
hr.new4 {
  border: 1px solid red;
}

/* Large rounded green border */
hr.new5 {
  border: 10px solid green;
  border-radius: 5px;
}

Then, to include a html, we have to use a w3-include-html attribute:

<div w3-include-html="content.html"></div>

Afterwards, we have to include a javascript:

<script>
function includeHTML() {
  var z, i, elmnt, file, xhttp;
  /* Loop through a collection of all HTML elements: */
  z = document.getElementsByTagName("*");
  for (i = 0; i < z.length; i++) {
    elmnt = z[i];
    /*search for elements with a certain atrribute:*/
    file = elmnt.getAttribute("w3-include-html");
    if (file) {
      /* Make an HTTP request using the attribute value as the file name: */
      xhttp = new XMLHttpRequest();
      xhttp.onreadystatechange = function() {
        if (this.readyState == 4) {
          if (this.status == 200) {elmnt.innerHTML = this.responseText;}
          if (this.status == 404) {elmnt.innerHTML = "Page not found.";}
          /* Remove the attribute, and call this function once more: */
          elmnt.removeAttribute("w3-include-html");
          includeHTML();
        }
      }
      xhttp.open("GET", file, true);
      xhttp.send();
      /* Exit the function: */
      return;
    }
  }
}
</script>

Then at the bottom of the page, we call the includeHTML() function:

<script>
includeHTML();
</script>

Designs

We can ignore the body part of the page, as it is handled directly by Pandoc. However, the site should have a navbar, and a footer. The original inspiration is write.as layout, as it is simple.

From that inspiration, I want to keep its simplicity, hence reduction of useless elements. We want only to read the content, and easy navigation. We however, want to make sure that it is also not ugly.

Hence, it starts with a header, that would contain a neat navbar. It should also point to a homescreen, or rather, an index page. The navbar should point to our current sites, the list of which should be:

  • xenostation: leads to the index page.
  • wiki: leads to the traditional xenomancy wiki.
  • uploads: leads to an index for upload files, or SBA-002.
  • about: leads to SBA-001.

The index page should be located at a root folder. Then the entries, would be primarily listed at the index page. Since we’re doing serialized entry numbers, it should ease hyperlinking. We don’t have to match the page name with the file name, just numbers.

Each page other than the root index page is stored in a pages folder, and is serialized. For example, this article with its ticker SBA-000, is stored in /pages/SBA-000.html. We can do the same for the rest of Series I (item SBA-000 thru SBA-999.

By design, there must be no other page with names but the /index.html page. Therefore, we have to fit our notes and entries into one of the SBA entries. It also applies for our about page, or other related pages.

Due to the nature of the permaweb, and manifest file of ArDrive, the version you’re reading depends on the manifest file you’re accessing this blog from. To test it, click on the xenoashton button at the navbar above, and check its date. As a rule of thumb, the date of the index page tells you which version of the blog are you seeing. Newer dates mean recent versions.

Following the header, navbar, and basic directory structures, is the body of the blog. It is the content, and it can be anything we like. All contents would start their life as a markdown document, parsed by pandoc into a html page. Whatever is allowed in a pandoc markdown file, is going to find its life in the body. One tiny modification we are going to use is that we are going to justify the content in the styling definition on our header.

Then comes the footer. The footer would also be equally simple. It will consist of just a horizontal line, and a centered text. The text would be xenostation blog by Hendrik Lie, with a link to our normal web resume page.

After the design is done, we can start describe the construction. However, it is worth noting that we should only deploy the website until we at least have a working primitive. That is, the index page should be live, along with the uploads page, and the about page. Other pages might follow later.

Now, there are plenty of files to be included. We also need to specify certain elements through pandoc. It might mess up with parsing if we didn’t do it right. Typing a long command with its parameters is prone of error. We have to make it work flawlessly every single time.

Hence we use the pandoc defaults file static.yaml:

---
standalone: true
toc: true
toc-depth: 3
title-prefix: xenoblog
include-in-header:
- ${.}/snips/favicon.html
include-before-body:
- ${.}/snips/header.html
include-after-body:
- ${.}/snips/footer.html
html-math-method:
  method: mathjax
metadata-file: meta.yaml
verbosity: INFO

The file handles all necessary pandoc arguments, hence when we have to compile our markdown notes, we just have to type:

pandoc -d static input.md -o output.html

Note that -d static argument ignores the .yaml extension of our static.yaml file.

It includes our favicon, header, and footer. It also prepares the need of math equations in the body, by specifying mathjax method. toc is set to be true, so we can always have table of contents in our pages. title-prefix allows us to have the page to produce our blog name: xenoblog

Update M809.2136

I feel stupid. ~/index.html resolved into the gateway’s root folder, which of course wouldn’t work. In another word, my previous assumption that the root folder of a manifest file would be the level of the manifest file itself. It is wrong, it is the gateway’s root folder instead. Therefore we can only use relative links.

From HTML File Paths we know that:

Path Description
<img src="picture.jpg"> The “picture.jpg” file is located in the same folder as the current page
<img src="images/picture.jpg"> The “picture.jpg” file is located in the images folder in the current folder
<img src="/images/picture.jpg"> The “picture.jpg” file is located in the images folder at the root of the current web
<img src="../picture.jpg"> The “picture.jpg” file is located in the folder one level up from the current folder

Therefore the structure should be:

  1. For index.html, the links should refer to folders in the current folder.
  2. For SBA-XXX.html files, links and resources should refer to a location one level up from the current folder.

It simply means that we should have two different navbar, one for index.html and another for other pages. The one for index file should behave as if all resources are on the same level or under its own level. The one for everything else should behave as if all other pages are on the same level as it is except index page, and resources (images, etc) are on a location one level up from their current location.

To implement that, we simply have two navbar files, and two defaults files.

Navbar for index page navbar-index.html would be:

<nav>
<ul class="nav">
  <li class="nav"><a href="index.html">xenostation</a></li>
  <li class="nav"><a href="https://xenomancy.id/">wiki</a></li>
  <li class="nav"><a href="pages/SBA-002.html">uploads</a></li>
  <li class="nav" style="float:right"><a href="pages/SBA-001.html">about</a></li>
</ul>
</nav>

Navbar for other pages navbar.html would be:

<nav>
<ul class="nav">
  <li class="nav"><a href="../index.html">xenostation</a></li>
  <li class="nav"><a href="https://xenomancy.id/">wiki</a></li>
  <li class="nav"><a href="../pages/SBA-002.html">uploads</a></li>
  <li class="nav" style="float:right"><a href="../pages/SBA-001.html">about</a></li>
</ul>
</nav>

The header.html would then only contain styling elements.

Meanwhile the index.html page would refer contents on the same level as it is. So instead of:

[SBA-001](/pages/SBA-001.html)

We do:

[SBA-001](pages/SBA-001.html)

Then we make static.yaml and static-index.yaml, where the former would refer to navbar.html on include-before-body, and the later would instead refer to navbar-index.html on include-before-body.
